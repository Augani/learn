# Lesson 5: Recursive Descent Parsing ‚Äî Turning Tokens Into Trees

The lexer gave us a flat stream of tokens. Now we need to build a tree. The parser reads
tokens left to right and figures out the hierarchical structure of the program ‚Äî which
expressions are nested inside which statements, which operands belong to which operators,
and where one construct ends and another begins.

Recursive descent is the most intuitive way to do this. It's what Go's own compiler uses
(`go/parser`), what V8 uses for JavaScript, and what you'll find in most hand-written
parsers in production. It's top-down, predictable, and maps directly to your grammar.

---

## The Sentence-Reading Analogy

Imagine you're reading English text aloud and someone asks you to diagram the sentence.

You see the first word: "The". You immediately know a noun phrase is starting. You keep
reading ‚Äî "The big red dog" ‚Äî until you hit a verb. Now you know the noun phrase is done
and a verb phrase is beginning. "The big red dog chased the cat across the yard."

You parsed that top-down. You started with "this is a sentence" and broke it into a noun
phrase and a verb phrase. Then you broke the noun phrase into an article, adjectives, and a
noun. You broke the verb phrase into a verb, an object (another noun phrase), and a
prepositional phrase. Each time you looked at the next word to decide which sub-rule to
apply.

That's exactly what a recursive descent parser does with tokens.

```
Sentence: "The big red dog chased the cat across the yard."

         Sentence
        /        \
   NounPhrase    VerbPhrase
   /  |   |  \     /    |      \
 The big red dog  chased NounPhrase  PrepPhrase
                         /    \      /     \
                       the    cat  across  NounPhrase
                                           /     \
                                         the    yard
```

Replace "noun phrase" with "expression" and "verb phrase" with "statement," and you have a
programming language parser. Clauses within sentences are parsed recursively ‚Äî a noun
phrase can contain a prepositional phrase that itself contains another noun phrase. Same
idea: an expression can contain a function call whose arguments are themselves expressions.

---

## Top-Down vs Bottom-Up

There are two families of parsing strategies:

| Strategy     | Direction     | How it works                                       | Used by            |
|-------------|---------------|----------------------------------------------------|--------------------|
| **Top-down**  | Start ‚Üí leaves | Start with "program" and break into smaller parts  | Recursive descent, LL parsers |
| **Bottom-up** | Leaves ‚Üí start | Start with individual tokens and combine upward     | yacc, bison, LR parsers |

Recursive descent is **top-down**. You start by calling `ParseProgram()`, which calls
`ParseStatement()`, which might call `ParseLetStatement()`, which calls
`ParseExpression()`, which might call `ParseInfixExpression()`, and so on. Each function
call goes deeper into the tree.

Bottom-up parsers (like the ones generated by yacc) work in reverse ‚Äî they read tokens,
accumulate them on a stack, and figure out what rule they match after the fact. They're
powerful but harder to write by hand and harder to debug.

We're going top-down because:
1. Every grammar rule maps to one function ‚Äî easy to understand and modify
2. Error messages are excellent ‚Äî you always know what you expected
3. It's what production languages actually use

---

## Grammar Rules Become Functions

This is the core insight. Given a grammar like:

```
Program        ‚Üí Statement*
Statement      ‚Üí LetStatement | ReturnStatement | ExpressionStatement
LetStatement   ‚Üí "let" Identifier "=" Expression ";"
ReturnStatement ‚Üí "return" Expression ";"
ExpressionStatement ‚Üí Expression ";"
```

Each rule becomes a Go function:

```go
func (p *Parser) ParseProgram() *ast.Program {
	program := &ast.Program{}

	for p.currentToken.Type != token.EOF {
		stmt := p.parseStatement()
		if stmt != nil {
			program.Statements = append(program.Statements, stmt)
		}
		p.nextToken()
	}

	return program
}

func (p *Parser) parseStatement() ast.Statement {
	switch p.currentToken.Type {
	case token.LET:
		return p.parseLetStatement()
	case token.RETURN:
		return p.parseReturnStatement()
	default:
		return p.parseExpressionStatement()
	}
}

func (p *Parser) parseLetStatement() *ast.LetStatement {
	stmt := &ast.LetStatement{Token: p.currentToken}

	if !p.expectPeek(token.IDENT) {
		return nil
	}

	stmt.Name = &ast.Identifier{
		Token: p.currentToken,
		Value: p.currentToken.Literal,
	}

	if !p.expectPeek(token.ASSIGN) {
		return nil
	}

	p.nextToken()
	stmt.Value = p.parseExpression(LOWEST)

	if p.peekTokenIs(token.SEMICOLON) {
		p.nextToken()
	}

	return stmt
}
```

See the pattern? The grammar rule `LetStatement ‚Üí "let" Identifier "=" Expression ";"`
became a function that:
1. Checks for `let` (it was already matched by the caller's switch)
2. Expects an identifier next
3. Expects `=` next
4. Parses an expression
5. Expects `;` at the end

Each expectation either succeeds (and advances the token) or fails (and reports an error).
The recursion comes from step 4 ‚Äî `parseExpression` will call other parse functions that
might call `parseExpression` again for nested expressions.

---

## The Parser's State Machine

A recursive descent parser is really just a state machine that walks through the token
stream. It needs to track two things:

```go
type Parser struct {
	lexer        *lexer.Lexer
	currentToken token.Token
	peekToken    token.Token
	errors       []string
}
```

**`currentToken`** is the token we're currently examining. **`peekToken`** is the next
token in the stream ‚Äî we haven't consumed it yet, but we can look at it to decide what to
do. This is like reading a sentence and glancing at the next word before deciding how to
parse the current phrase.

The two fundamental operations:

```go
func (p *Parser) nextToken() {
	p.currentToken = p.peekToken
	p.peekToken = p.lexer.NextToken()
}

func (p *Parser) expectPeek(t token.TokenType) bool {
	if p.peekToken.Type == t {
		p.nextToken()
		return true
	}
	p.peekError(t)
	return false
}
```

`nextToken()` advances the cursor by one position. `expectPeek()` is a guarded advance ‚Äî
it only moves forward if the next token is what we expect. If it's not, we record an error.
This is the heart of predictive parsing: peek ahead, make a decision, then consume.

---

## Predictive Parsing ‚Äî LL(1)

The "(1)" in LL(1) means we look **one token ahead** to decide which grammar rule to
apply. The two L's stand for:
- **L**eft-to-right scanning of input
- **L**eftmost derivation (we always expand the leftmost non-terminal first)

In practice, LL(1) means: when we're inside `parseStatement()`, we look at `currentToken`
and that single token tells us which statement type to parse.

```go
func (p *Parser) parseStatement() ast.Statement {
	switch p.currentToken.Type {
	case token.LET:
		return p.parseLetStatement()
	case token.RETURN:
		return p.parseReturnStatement()
	default:
		return p.parseExpressionStatement()
	}
}
```

One token is enough to decide. If we see `let`, it's a let statement. If we see `return`,
it's a return statement. Everything else is an expression statement. No ambiguity.

This works because we designed the grammar to be **unambiguous at the first token**. Each
statement type starts with a unique keyword. Not all grammars have this property ‚Äî some
need more lookahead (LL(2), LL(k)) or can't be parsed top-down at all. But most practical
programming languages are designed to be LL(1) or close to it.

What happens when one token isn't enough? Consider parsing expressions. When you see the
token `5`, is it:
- An integer literal: `5`
- The start of an addition: `5 + 3`
- The start of a function call: `5()` (weird but syntactically possible)

You can't tell from one token. This is where Pratt parsing saves us.

---

## The Expression Problem

Statements are easy to parse top-down. Expressions are hard. The difficulty comes from
three properties of expressions:

**1. Operator precedence**: `1 + 2 * 3` must parse as `1 + (2 * 3)`, not `(1 + 2) * 3`.
The parser needs to know that `*` binds tighter than `+`.

**2. Associativity**: `1 - 2 - 3` must parse as `(1 - 2) - 3` (left-associative), not
`1 - (2 - 3)`. Without this, subtraction would give wrong results.

**3. Prefix vs infix**: The `-` in `-5` is a prefix operator (it appears before its
operand). The `-` in `10 - 5` is an infix operator (it appears between two operands). Same
token, different meaning depending on position.

A naive recursive descent approach would need a separate function for each precedence level:

```
parseAddition      ‚Üí parseMultiplication ( ("+" | "-") parseMultiplication )*
parseMultiplication ‚Üí parseUnary ( ("*" | "/") parseUnary )*
parseUnary         ‚Üí ("-" | "!") parseUnary | parsePrimary
parsePrimary       ‚Üí INTEGER | IDENTIFIER | "(" Expression ")"
```

This works, but it's rigid. Every time you add a precedence level, you add another
function. Languages with 15+ precedence levels (like C) would need 15+ functions, each one
calling the next. Pratt parsing eliminates this entirely.

## Pratt Parsing ‚Äî Precedence Climbing

Vaughan Pratt published this technique in 1973. It's elegant: instead of encoding
precedence in the call stack (one function per level), you encode it as a **number** ‚Äî the
binding power. One function handles all precedence levels.

### The Core Idea: Binding Power

Think of operators as magnets with different strengths. In `1 + 2 * 3`:

```
    1   +   2   *   3
        ^       ^
       weak   strong
```

The `*` magnet is stronger than `+`, so it pulls `2` and `3` toward itself first:

```
    1   +   (2 * 3)
```

We assign each operator a **binding power** (a number). Higher number = stronger pull:

```go
const (
	LOWEST      = iota
	EQUALS
	LESSGREATER
	SUM
	PRODUCT
	PREFIX
	CALL
	INDEX
)
```

| Precedence Level | Value | Operators          |
|-----------------|-------|--------------------|
| LOWEST          | 0     | (starting value)   |
| EQUALS          | 1     | `==`, `!=`         |
| LESSGREATER     | 2     | `<`, `>`           |
| SUM             | 3     | `+`, `-`           |
| PRODUCT         | 4     | `*`, `/`           |
| PREFIX          | 5     | `-x`, `!x`         |
| CALL            | 6     | `fn()`             |
| INDEX           | 7     | `arr[0]`           |

The rule: an operator can only claim an operand if its binding power is **higher than** the
binding power of the operator currently trying to claim that operand. This is what creates
the correct tree structure.

### Prefix and Infix Parse Functions

Pratt parsing divides parsing into two categories:

**Prefix parse functions** handle tokens that appear at the **start** of an expression:
- Integer literals: `42`
- Identifiers: `foo`
- Prefix operators: `-5`, `!true`
- Grouped expressions: `(1 + 2)`
- If expressions: `if (x > 5) { ... }`
- Function literals: `fn(x, y) { ... }`

**Infix parse functions** handle tokens that appear **between** two expressions (the left
side has already been parsed):
- Arithmetic: `a + b`, `a * b`
- Comparison: `a == b`, `a < b`
- Call expressions: `add(1, 2)` ‚Äî yes, `(` is an infix operator where the left side is the
  function name
- Index expressions: `arr[0]` ‚Äî `[` is an infix operator

We register these in lookup tables:

```go
type (
	prefixParseFn func() ast.Expression
	infixParseFn  func(ast.Expression) ast.Expression
)

type Parser struct {
	lexer        *lexer.Lexer
	currentToken token.Token
	peekToken    token.Token
	errors       []string

	prefixParseFns map[token.TokenType]prefixParseFn
	infixParseFns  map[token.TokenType]infixParseFn
}
```

Registration during parser construction:

```go
func New(l *lexer.Lexer) *Parser {
	p := &Parser{lexer: l}

	p.prefixParseFns = make(map[token.TokenType]prefixParseFn)
	p.registerPrefix(token.IDENT, p.parseIdentifier)
	p.registerPrefix(token.INT, p.parseIntegerLiteral)
	p.registerPrefix(token.BANG, p.parsePrefixExpression)
	p.registerPrefix(token.MINUS, p.parsePrefixExpression)
	p.registerPrefix(token.TRUE, p.parseBoolean)
	p.registerPrefix(token.FALSE, p.parseBoolean)
	p.registerPrefix(token.LPAREN, p.parseGroupedExpression)
	p.registerPrefix(token.IF, p.parseIfExpression)
	p.registerPrefix(token.FUNCTION, p.parseFunctionLiteral)

	p.infixParseFns = make(map[token.TokenType]infixParseFn)
	p.registerInfix(token.PLUS, p.parseInfixExpression)
	p.registerInfix(token.MINUS, p.parseInfixExpression)
	p.registerInfix(token.SLASH, p.parseInfixExpression)
	p.registerInfix(token.ASTERISK, p.parseInfixExpression)
	p.registerInfix(token.EQ, p.parseInfixExpression)
	p.registerInfix(token.NOT_EQ, p.parseInfixExpression)
	p.registerInfix(token.LT, p.parseInfixExpression)
	p.registerInfix(token.GT, p.parseInfixExpression)
	p.registerInfix(token.LPAREN, p.parseCallExpression)

	p.nextToken()
	p.nextToken()

	return p
}
```

### The Heart of Pratt Parsing: `parseExpression`

This is the single most important function. It's about 15 lines of code and it handles
every possible expression at every possible precedence level:

```go
func (p *Parser) parseExpression(precedence int) ast.Expression {
	prefix := p.prefixParseFns[p.currentToken.Type]
	if prefix == nil {
		p.noPrefixParseFnError(p.currentToken.Type)
		return nil
	}

	leftExp := prefix()

	for !p.peekTokenIs(token.SEMICOLON) && precedence < p.peekPrecedence() {
		infix := p.infixParseFns[p.peekToken.Type]
		if infix == nil {
			return leftExp
		}

		p.nextToken()
		leftExp = infix(leftExp)
	}

	return leftExp
}
```

Read it step by step:

1. **Find a prefix parser** for the current token. If the token is `5`, the prefix parser
   is `parseIntegerLiteral`. If it's `-`, it's `parsePrefixExpression`. If there's no
   prefix parser registered for this token type, it's a syntax error.

2. **Call the prefix parser** to get the left-hand expression.

3. **Loop**: while the next token has higher precedence than our current level, consume it
   as an infix operator. This is where precedence climbing happens ‚Äî we keep going as long
   as the next operator binds tighter.

4. **Return** the accumulated expression.

The magic is in the loop condition: `precedence < p.peekPrecedence()`. The `precedence`
parameter is the binding power of the calling context. The loop continues only while the
next operator is stronger. When we hit an operator that's weaker or equal, we stop and
return ‚Äî letting the caller handle it.

## Walkthrough: Parsing `1 + 2 * 3`

Let's trace through the algorithm step by step. The tokens are:

```
[INT "1"] [PLUS "+"] [INT "2"] [ASTERISK "*"] [INT "3"] [EOF]
```

**Step 1**: `ParseProgram` calls `parseExpressionStatement` which calls
`parseExpression(LOWEST)`.

```
parseExpression(precedence=0)
  currentToken = INT "1"
```

**Step 2**: Look up prefix parser for `INT`. It's `parseIntegerLiteral`. Call it.

```go
func (p *Parser) parseIntegerLiteral() ast.Expression {
	value, err := strconv.ParseInt(p.currentToken.Literal, 0, 64)
	if err != nil {
		p.errors = append(p.errors, fmt.Sprintf("could not parse %q as integer", p.currentToken.Literal))
		return nil
	}
	return &ast.IntegerLiteral{Token: p.currentToken, Value: value}
}
```

Now `leftExp = IntegerLiteral(1)`.

**Step 3**: Check the loop condition. `peekToken` is `PLUS`. The precedence of `PLUS` is
`SUM` (3). Is `0 < 3`? Yes. Enter the loop.

**Step 4**: Look up infix parser for `PLUS`. It's `parseInfixExpression`. Advance token.
Call it with `leftExp = IntegerLiteral(1)`:

```go
func (p *Parser) parseInfixExpression(left ast.Expression) ast.Expression {
	expression := &ast.InfixExpression{
		Token:    p.currentToken,
		Operator: p.currentToken.Literal,
		Left:     left,
	}

	precedence := p.currentPrecedence()
	p.nextToken()
	expression.Right = p.parseExpression(precedence)

	return expression
}
```

Inside `parseInfixExpression`:
- `left = IntegerLiteral(1)`
- `operator = "+"`
- `precedence = SUM (3)`
- Advance token. `currentToken` is now `INT "2"`
- Call `parseExpression(3)` to get the right side

**Step 5**: Now we're in a **recursive call** to `parseExpression(precedence=3)`.

```
parseExpression(precedence=3)
  currentToken = INT "2"
```

Prefix parser for `INT` gives us `leftExp = IntegerLiteral(2)`.

**Step 6**: Check the loop condition. `peekToken` is `ASTERISK`. The precedence of
`ASTERISK` is `PRODUCT` (4). Is `3 < 4`? Yes. Enter the loop.

This is the critical moment. The `*` has higher precedence than the `+` that called us, so
`*` gets to claim `2` as its left operand.

**Step 7**: Look up infix parser for `ASTERISK`. Advance token. Call
`parseInfixExpression` with `leftExp = IntegerLiteral(2)`:

- `left = IntegerLiteral(2)`
- `operator = "*"`
- `precedence = PRODUCT (4)`
- Advance token. `currentToken` is now `INT "3"`
- Call `parseExpression(4)` to get the right side

**Step 8**: `parseExpression(precedence=4)`. Current token is `INT "3"`. Prefix parser
gives `IntegerLiteral(3)`. Peek is `EOF`. Precedence of `EOF` is `LOWEST` (0). Is `4 < 0`?
No. Exit loop. Return `IntegerLiteral(3)`.

**Step 9**: Back in Step 7's `parseInfixExpression`. Right side is `IntegerLiteral(3)`.
Return:

```
InfixExpression("*")
‚îú‚îÄ‚îÄ Left: IntegerLiteral(2)
‚îî‚îÄ‚îÄ Right: IntegerLiteral(3)
```

**Step 10**: Back in Step 5's `parseExpression(3)`. `leftExp` is now the `*` expression.
Check loop: peek is `EOF`, precedence 0. Is `3 < 0`? No. Exit loop. Return the `*`
expression.

**Step 11**: Back in Step 4's `parseInfixExpression`. Right side is the `*` expression.
Return:

```
InfixExpression("+")
‚îú‚îÄ‚îÄ Left: IntegerLiteral(1)
‚îî‚îÄ‚îÄ Right: InfixExpression("*")
           ‚îú‚îÄ‚îÄ Left: IntegerLiteral(2)
           ‚îî‚îÄ‚îÄ Right: IntegerLiteral(3)
```

**Step 12**: Back in Step 1's `parseExpression(0)`. `leftExp` is the full tree. Peek is
`EOF`. Exit loop. Return.

The final tree correctly represents `1 + (2 * 3)`:

```
         (+)
        /   \
       1    (*)
           /   \
          2     3
```

If `+` and `*` had the same precedence, the loop in Step 6 would NOT have entered (because
`3 < 3` is false), and `2` would have stayed as the right operand of `+`, giving us
`(1 + 2) * 3`. The precedence numbers are what enforce the correct grouping.

## Parsing Prefix Expressions

Prefix operators like `-5` or `!true` are handled by their own parse function:

```go
func (p *Parser) parsePrefixExpression() ast.Expression {
	expression := &ast.PrefixExpression{
		Token:    p.currentToken,
		Operator: p.currentToken.Literal,
	}

	p.nextToken()
	expression.Operand = p.parseExpression(PREFIX)

	return expression
}
```

The key detail: we call `parseExpression(PREFIX)` with a high precedence. This means the
operand of a prefix operator binds very tightly. `-a * b` parses as `(-a) * b`, not
`-(a * b)`, because the PREFIX precedence (5) is higher than PRODUCT (4). The prefix
operator grabs `a` before `*` gets a chance.

---

## Parsing Grouped Expressions

Parentheses override precedence. The parser handles them as a prefix:

```go
func (p *Parser) parseGroupedExpression() ast.Expression {
	p.nextToken()

	exp := p.parseExpression(LOWEST)

	if !p.expectPeek(token.RPAREN) {
		return nil
	}

	return exp
}
```

When we see `(`, we parse the inner expression starting at `LOWEST` precedence. This
resets the precedence floor ‚Äî everything inside the parentheses groups together before
anything outside gets a chance. Then we consume the closing `)`.

The result is that `(1 + 2) * 3` correctly parses as:

```
        (*)
       /   \
     (+)    3
    /   \
   1     2
```

The `+` claims both `1` and `2` because inside the parentheses, the precedence restarted
at LOWEST.

## Parsing Function Calls

Here's where it gets interesting. A function call like `add(1, 2)` is parsed as an infix
expression where `(` is the infix operator and `add` is the left operand:

```go
func (p *Parser) parseCallExpression(function ast.Expression) ast.Expression {
	exp := &ast.CallExpression{
		Token:    p.currentToken,
		Function: function,
	}
	exp.Arguments = p.parseExpressionList(token.RPAREN)
	return exp
}

func (p *Parser) parseExpressionList(end token.TokenType) []ast.Expression {
	var list []ast.Expression

	if p.peekTokenIs(end) {
		p.nextToken()
		return list
	}

	p.nextToken()
	list = append(list, p.parseExpression(LOWEST))

	for p.peekTokenIs(token.COMMA) {
		p.nextToken()
		p.nextToken()
		list = append(list, p.parseExpression(LOWEST))
	}

	if !p.expectPeek(end) {
		return nil
	}

	return list
}
```

This works because `(` is registered as an infix operator with `CALL` precedence. By the
time we reach the `(`, the parser has already parsed `add` as an identifier. Then the
infix handler receives that identifier as the `function` parameter and parses the argument
list.

This naturally handles chained calls like `getAdder(1)(2)` ‚Äî the first call expression
becomes the `function` argument of the second.

## Parsing Statements

Expression parsing is the hard part. Statements are straightforward because they start with
distinct keywords:

```go
func (p *Parser) parseReturnStatement() *ast.ReturnStatement {
	stmt := &ast.ReturnStatement{Token: p.currentToken}

	p.nextToken()
	stmt.ReturnValue = p.parseExpression(LOWEST)

	if p.peekTokenIs(token.SEMICOLON) {
		p.nextToken()
	}

	return stmt
}

func (p *Parser) parseExpressionStatement() *ast.ExpressionStatement {
	stmt := &ast.ExpressionStatement{Token: p.currentToken}

	stmt.Expression = p.parseExpression(LOWEST)

	if p.peekTokenIs(token.SEMICOLON) {
		p.nextToken()
	}

	return stmt
}
```

The `parseExpressionStatement` is the default case ‚Äî when we don't see a keyword like `let`
or `return`, we assume the entire statement is an expression. This is how standalone
expressions like `add(1, 2);` or `x + y;` get parsed.

## Error Recovery

A production parser needs to recover from errors and keep going, so it can report multiple
issues at once instead of dying at the first problem. The simplest recovery strategy is
**synchronization** ‚Äî when you hit an error, skip tokens until you find a safe point to
resume (usually a semicolon or a keyword like `let`):

```go
func (p *Parser) peekError(t token.TokenType) {
	msg := fmt.Sprintf("expected next token to be %s, got %s instead",
		t, p.peekToken.Type)
	p.errors = append(p.errors, msg)
}

func (p *Parser) noPrefixParseFnError(t token.TokenType) {
	msg := fmt.Sprintf("no prefix parse function for %s found", t)
	p.errors = append(p.errors, msg)
}
```

Errors are collected in a slice rather than causing an immediate panic. After parsing
completes, the caller checks `parser.Errors()` and reports them all.

## Go's Own Parser

Go's standard library includes `go/parser`, which parses Go source code into ASTs. It uses
recursive descent ‚Äî the same technique we've been discussing. You can inspect it yourself:

```go
package main

import (
	"fmt"
	"go/ast"
	"go/parser"
	"go/token"
)

func main() {
	src := `package main

func main() {
	x := 1 + 2*3
}
`
	fset := token.NewFileSet()
	file, _ := parser.ParseFile(fset, "example.go", src, 0)
	ast.Print(fset, file)
}
```

Running this prints the full AST that Go's parser produces. If you dig into Go's source
code (`src/go/parser/parser.go`), you'll see familiar patterns:

- `parseStmt()` dispatches to specific statement parsers based on the current token
- `parseExpr()` handles expressions with precedence
- `expect()` and `next()` manage token advancement
- Error recovery uses synchronization to statement boundaries

The main difference from our Pratt approach is that Go's parser uses the classical
precedence-climbing style with explicit loops for each precedence level rather than lookup
tables of parse functions. Both approaches are forms of recursive descent ‚Äî they just
handle precedence differently.

## Putting It All Together

Here's how the full parser structure fits together:

```
ParseProgram()
  ‚îî‚îÄ‚îÄ parseStatement()         (switch on current token)
        ‚îú‚îÄ‚îÄ parseLetStatement()
        ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(LOWEST)
        ‚îú‚îÄ‚îÄ parseReturnStatement()
        ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(LOWEST)
        ‚îî‚îÄ‚îÄ parseExpressionStatement()
              ‚îî‚îÄ‚îÄ parseExpression(precedence)
                    ‚îú‚îÄ‚îÄ prefix fn ‚Üí leftExp
                    ‚îÇ     ‚îú‚îÄ‚îÄ parseIdentifier()
                    ‚îÇ     ‚îú‚îÄ‚îÄ parseIntegerLiteral()
                    ‚îÇ     ‚îú‚îÄ‚îÄ parsePrefixExpression()
                    ‚îÇ     ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(PREFIX)
                    ‚îÇ     ‚îú‚îÄ‚îÄ parseGroupedExpression()
                    ‚îÇ     ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(LOWEST)
                    ‚îÇ     ‚îú‚îÄ‚îÄ parseIfExpression()
                    ‚îÇ     ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(LOWEST)
                    ‚îÇ     ‚îî‚îÄ‚îÄ parseFunctionLiteral()
                    ‚îÇ           ‚îî‚îÄ‚îÄ parseBlockStatement()
                    ‚îî‚îÄ‚îÄ infix loop (while precedence < peek precedence)
                          ‚îú‚îÄ‚îÄ parseInfixExpression(left)
                          ‚îÇ     ‚îî‚îÄ‚îÄ parseExpression(currentPrecedence)
                          ‚îî‚îÄ‚îÄ parseCallExpression(left)
                                ‚îî‚îÄ‚îÄ parseExpressionList()
```

Every call flows top-down. The recursion happens when an expression contains
sub-expressions: a function call's arguments are expressions, an if-expression's condition
is an expression, a binary operator's right-hand side is an expression. The precedence
parameter threads through the recursion, ensuring operators bind correctly at every level.

---

## Key Takeaways

**Recursive descent is a direct mapping from grammar to code.** Each grammar rule becomes a
function. The function looks at the current token, decides what to parse, consumes tokens,
builds an AST node, and returns it. The recursion handles nesting naturally.

**Pratt parsing solves the expression problem.** Instead of one function per precedence
level, you have one function (`parseExpression`) that uses binding power numbers to decide
how tightly operators hold their operands. Prefix and infix parse functions are registered
in lookup tables, making the parser easy to extend.

**The precedence parameter is the key mechanism.** When `parseExpression` is called with a
precedence of `SUM`, it will consume operators with precedence higher than `SUM` (like
`PRODUCT`) but stop at operators with equal or lower precedence. This creates the correct
tree structure without any explicit precedence-level functions.

**One token of lookahead is almost always enough.** LL(1) parsing ‚Äî peeking at the next
token to decide what rule to apply ‚Äî works for the vast majority of programming language
constructs. The few cases where it doesn't (like distinguishing `a` from `a(b)`) are
handled elegantly by Pratt parsing's infix table.
